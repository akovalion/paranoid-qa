## 5. Artifacts and bug reports

- **Evidence.** Minimal sufficient set, each artifact confirms a specific step. Before/after screenshots with the problem highlighted; screenshot with DevTools open (Console/Network) for frontend errors; full-page for layout/scroll bugs; URL bar and resolution visible in the frame. Video/GIF for flaky/multi-step/race bugs (start 1-2 steps before the problem, 10-40 sec). Mask PII/tokens/payment details on all attachments.
- **Network/server.** HAR (with "Preserve log", scrubbed of secrets); for an API bug — method+URL, request/response headers/body, status, timing; requestId/traceId/correlationId to correlate with logs; exact reproduction time and TZ; link to Sentry/trace; for a frontend/backend mismatch — what the API returned vs what the UI rendered.
- **Console.** Full stack trace (not just the header line); levels (error→bug, warning separately); reproducibility in a clean session; hydration/CORS/CSP/mixed-content errors as a separate class.
- **Bug report structure.** Title `[Section] essence` (what/where/under which condition, readable without opening the body, not "doesn't work"); preconditions (state, role, data, flags); steps (numbered, atomic, from a clean state, reproducible by an outsider); one bug = one report; frequency (100% / N of M); actual result (with a link to the attachment); expected (with a link to the source of truth — mockup/AC/spec, not "by common sense"); problem layer (frontend vs backend).
- **Environment.** dev/stage/prod + build/commit; browser+version, OS; for responsive — device/emulation + resolution; desktop vs mobile explicitly; network/throttling/locale/TZ if they matter; account/role (no password); zoom ≠ 100% (a frequent cause of "floating" layout).
- **Severity vs Priority.** Severity — technical impact (blocker/critical/major/minor/trivial), set by QA; Priority — business urgency, set by the PO. Do not conflate: high severity + low priority is valid, and vice versa. Blocker — no workaround, breaks the flow/testing; critical — core functionality unavailable but a workaround exists. Justify severity by impact (users/frequency/workaround), not by feel.
- **Mockup discrepancies → question to the analyst.** A discrepancy with Figma under an ambiguous requirement is a question, not immediately a bug; present in the mockup / absent in the implementation without AC → clarification; side-by-side (mockup fragment vs implementation with annotation); link to the specific frame/node with version; states missing from the mockup (errors/empty/loading/overflow) — a separate question; copy/microcopy verbatim, a text discrepancy is a separate category. Distinguish: mockup violation (bug) vs case not covered by the mockup (gap → to the analyst).
- **Tracker/TMS (Jira, Zephyr, TestRail, etc.).** Link the defect to the test case and the ticket (blocks/relates/is caused by); test case statuses (Pass/Fail/Blocked/Not Executed/In Progress honestly, Blocked ≠ Fail); Fail always with a defect, Blocked with a reason; test case and bug in the same cycle/version (Affects/Fix Version, component, labels); reopen the existing issue instead of a duplicate (search for duplicates); after the fix — retest by the steps + regression around it, close as "verified on build X".
- **Run summary report.** Header (release/build/environment, when, by whom, type SMOKE/REGRESS/API); metrics (total test cases, Pass/Fail/Blocked/Skipped, pass rate %, defects by severity); failed test cases with links to defects; blockers/risks/uncovered areas/deliberate skips; flaky separately (passed on the Nth attempt, do not present as Pass); go/no-go verdict with rationale (QA recommendation); for automated tests — link to the HTML report/Allure/CI + trace.zip for failures; delta vs the previous run (new vs known); separate the cause (product defect vs test/environment/data issue).
- **Bug report edge cases.** Bug only on a repeated action/after a reroll (exact starting condition); only on specific data (attach the data); only in a specific order (state the sequence precisely); depends on cache/cookies/session (state whether a clean session is required); timing-dependent (video mandatory); only on one environment (put it in the title); a "green" screenshot for critical checks as evidence; verify the attachment actually opens; won't fix/by design — link it, don't file anew.
