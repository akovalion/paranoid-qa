## 5. Артефакты и баг-репорты

- **Доказательства.** Минимально достаточный набор, каждый артефакт подтверждает конкретный шаг. Скриншоты «до/после» с подсветкой проблемы; скриншот с открытым DevTools (Console/Network) для фронтовых ошибок; full-page для багов вёрстки/скролла; URL-бар и разрешение видны в кадре. Видео/GIF для плавающих/многошаговых/race-багов (за 1–2 шага до проблемы, 10–40 сек). Маскировать ПДн/токены/реквизиты на всех вложениях.
- **Сетевые/серверные.** HAR (с «Preserve log», очищенный от секретов); для API-бага — метод+URL, request/response headers/body, статус, время; requestId/traceId/correlationId для связки с логами; точное время и TZ воспроизведения; ссылка на Sentry/трейс; для расхождения фронт/бэк — что вернул API против того, что отрисовал UI.
- **Консоль.** Полный stack trace (не один заголовок); уровни (error→баг, warning отдельно); воспроизводимость в чистой сессии; ошибки гидратации/CORS/CSP/mixed-content отдельным классом.
- **Структура баг-репорта.** Заголовок `[Раздел] суть` (что/где/при каком условии, читается без открытия тела, не «не работает»); предусловия (состояние, роль, данные, флаги); шаги (пронумерованы, атомарны, от чистого состояния, воспроизводимы посторонним); один баг = один репорт; частота (100% / N из M); фактический результат (со ссылкой на вложение); ожидаемый (со ссылкой на источник истины — макет/AC/спека, не «по логике»); слой проблемы (фронт vs бэк).
- **Окружение.** dev/stage/prod + билд/commit; браузер+версия, ОС; для адаптива — устройство/эмуляция + разрешение; десктоп vs мобильный явно; сеть/троттлинг/локаль/TZ если влияют; учётка/роль (без пароля); zoom ≠ 100% (частая причина «плавающей» вёрстки).
- **Severity vs Priority.** Severity — техническое влияние (блокер/критич/мажор/минор/тривиал), ставит QA; Priority — срочность для бизнеса, ставит PO. Не путать: высокий severity + низкий priority валиден и наоборот. Блокер — нет обхода, ломает флоу/тестирование; критич — основной функционал недоступен, но есть workaround. Обосновывать severity влиянием (пользователи/частота/обход), не ощущением.
- **Расхождения с макетом → вопрос аналитику.** Расхождение с Figma при неоднозначном требовании — вопрос, не сразу баг; есть в макете/нет в реализации без AC → уточнение; side-by-side (фрагмент макета vs реализация с пометкой); ссылка на конкретный фрейм/ноду с версией; состояния, которых нет в макете (ошибки/пустые/загрузка/overflow) — отдельный вопрос; копирайт/микротексты дословно, расхождение текста — отдельная категория. Различать: нарушение макета (баг) vs непокрытый макетом кейс (gap → аналитику).
- **Трекер/TMS (Jira, Zephyr, TestRail и др.).** Связать дефект с ТК и тикетом (blocks/relates/is caused by); статусы ТК (Pass/Fail/Blocked/Not Executed/In Progress честно, Blocked ≠ Fail); Fail обязательно с дефектом, Blocked с причиной; ТК и баг в одном цикле/версии (Affects/Fix Version, компонент, лейблы); reopen существующего вместо дубля (искать дубликаты); после фикса — ретест по шагам + regress вокруг, close «verified on build X».
- **Сводный отчёт прогона.** Шапка (релиз/билд/окружение, когда, кем, тип SMOKE/REGRESS/API); метрики (всего ТК, Pass/Fail/Blocked/Skipped, % прохождения, дефекты по severity); упавшие ТК со ссылками на дефекты; блокеры/риски/непокрытое/осознанные пропуски; flaky отдельно (прошёл с N-й попытки, не выдавать за Pass); вердикт go/no-go с обоснованием (рекомендация QA); для автотестов — ссылка на HTML report/Allure/CI + trace.zip по упавшим; дельта к прошлому прогону (новые vs известные); разделять причину (дефект продукта vs проблема теста/окружения/данных).
- **Edge баг-репортов.** Баг только при повторном действии/после реролла (точное условие старта); только на конкретных данных (приложить данные); только при определённом порядке (последовательность точно); зависит от кэша/куки/сессии (нужна ли чистая сессия); с таймингом (видео обязательно); только на одном окружении (в заголовок); скриншот «зелёного» для критичных как доказательство; проверить, что вложение реально открывается; won't fix/by design — линковать, не заводить заново.
