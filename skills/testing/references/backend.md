## 3. Backend / Сервисы

### 3.1 API и контракты
- **Методы/эндпоинты:** корректный метод под операцию (GET без побочных эффектов); `405`+`Allow` на неподдерживаемый; OPTIONS/preflight, HEAD; регистр пути, trailing slash, URL-энкодинг (`%2F`, юникод); несуществующий → `404`, не `500`/HTML.
- **Коды статусов:** `200` vs `201`(+`Location`) vs `202`(async) vs `204`(реально без тела); `301/302/307/308` (сохранение метода/тела), `304`; `400`(валидация) vs `422`(семантика); `401` vs `403`; `404` vs `403` (не раскрывать существование); `409/410/412/415/429`; `5xx` не на ошибки клиента, без stacktrace/SQL/путей в теле; нет `200` с телом-ошибкой.
- **Схема/тело:** валидация request/response по OpenAPI/JSON Schema (типы, форматы, `required`, `enum`, `pattern`, `min/max`); форматы (ISO 8601 с TZ/offset, UUID, email, деньги — точность/округление), `null` vs отсутствие vs пустая строка; числовые границы (int32/int64/переполнение, отрицательные, дробные где ждут целое); вложенность/пустой `[]` vs `null`/дубликаты в массиве; кодировки (UTF-8, эмодзи, суррогаты, нулевой байт).
- **Поля:** отсутствие обязательного → понятная ошибка с указанием поля; неверный тип (`"5"` vs `5`, bool как `0/1`); опциональные с дефолтами; read-only/computed игнорируются при записи без `400`; чувствительные (пароли/токены/PAN) не возвращаются.
- **Лишние/неизвестные поля:** стратегия strict vs lenient зафиксирована; лишнее поле не даёт mass assignment (`isAdmin`/`roleId`/`price`); опечатка в имени ≠ пропуск обязательного.
- **Некорректный payload:** невалидный JSON → `400` не `500`; пустое тело/`{}`/пустой массив; дубль ключей; несоответствие `Content-Type` телу; глубокая вложенность/циклы (DoS-лимит); большой payload → `413`.
- **Контракты/совместимость:** drift detection (реализация = OpenAPI); consumer-driven (Pact); обратная совместимость (не удалять/переименовывать поля, не сужать типы, не менять enum без версии; добавление обязательного поля в request = breaking); версионирование (URL/header/media-type), `Deprecation`/`Sunset`.
- **Заголовки:** `Content-Type`+charset; `Accept`/content negotiation, `406`; кэш (`Cache-Control`/`ETag`+`If-None-Match`→304/`Vary`); CORS (origin/methods/headers/credentials, wildcard vs credentials); security (HSTS, nosniff, X-Frame-Options, CSP, нет `Server`/`X-Powered-By` с версиями); корреляция (`X-Request-Id` принимается/возвращается).
- **Пагинация/сортировка/фильтр/проекция:** offset/limit vs cursor; `limit=0`/отрицательный/сверх cap, offset за пределами → пустой список не ошибка; стабильность при вставке/удалении (дубли/пропуски), консистентность `total`/`hasNext`; сортировка по разрешённым полям, по неразрешённому → `400`/игнор, `null`-ordering; фильтры (операторы, AND/OR, инъекции через фильтр); проекция/sparse fieldsets (исключение чувствительных).
- **Идемпотентность:** GET/PUT/DELETE идемпотентны; повторный DELETE → `404`/`204` (зафиксировать); `Idempotency-Key` (повтор=тот же результат; тот же ключ+другое тело=конфликт; TTL); конкурентные одинаковые (один успех, нет дублей); optimistic locking (`If-Match`/version → `409`/`412`); retry после таймаута без двойного эффекта.
- **PATCH:** Merge Patch (`null`=удалить) vs JSON Patch (операции); PATCH несуществующего; пустой patch (no-op); read-only/immutable; не затирает не переданные поля; валидация результирующего объекта целиком.
- **Edge:** дублирующиеся query (`?id=1&id=2`), массивы в query; `null`/`""`/отсутствие/`0`/`false` в семантике; юникод-нормализация в id/поиске; ID > 2^53 как строка для JS; TZ/DST в фильтрах дат; двойной слэш/двойной URL-энкодинг/`414`; пустая коллекция → `200`+`[]` не `404`; gzip/br/декомпрессионная бомба; chunked без `Content-Length`; обрыв на середине.

### 3.2 Данные и БД
- **Персистентность CRUD:** записанное = возвращённое = реально в БД (проверять SELECT'ом, не только ответом API); Read возвращает сохранённое без служебных полей; Update меняет только указанные (`updated_at` обновился, `created_at` нет); Delete реально удалён/помечен, повторный идемпотентен, нет «зомби» в связанных; дефолты колонок; авто-поля (id/created_at/version); NULL vs `""` vs отсутствие ключа не теряются.
- **Целостность:** NOT NULL → ошибка не «тихий» дефолт; UNIQUE (составные, регистр, trailing-spaces, NULL в UNIQUE); FK (несуществующий родитель → отказ, удаление родителя с детьми); CHECK/доменные на уровне БД; каскады ON DELETE/UPDATE (нет неожиданного массового удаления); partial unique (только для активных).
- **Транзакции:** atomicity (сбой в середине → полный откат); откат при исключении/констрейнте/таймауте; savepoints; side-effects вне БД (письмо, событие) не при откате (transactional outbox); длинные транзакции (блокировки, рост WAL).
- **Изоляция/конкурентность:** уровни (READ COMMITTED/REPEATABLE READ/SERIALIZABLE) фактически vs ожидаемо; lost update; dirty/non-repeatable/phantom read; check-then-insert race → дубли (UNIQUE+upsert/`ON CONFLICT`); deadlock (retry, нет зависания); optimistic (version→409) vs пессимистичная (SELECT FOR UPDATE); конкурентный декремент остатка/баланса (нет минуса/двойного списания).
- **Миграции:** forward на чистой и заполненной БД (без потери данных); backward/rollback; idempotent/повторный запуск; совместимость со старым кодом (expand/contract, nullable колонка перед заполнением); backfill батчами без блокировки; индекс CONCURRENTLY; дрейф схемы (реальная БД = миграции, нет ручных правок).
- **Консистентность кэш/сервисы:** инвалидация кэша при записи (нет stale), TTL, cache stampede; dual-write (БД + событие атомарно через outbox/CDC); eventual consistency (read-after-write для своего запроса); денормализованные проекции обновляются; saga/компенсации.
- **Soft-delete:** удалённые не в выборках/поиске/агрегатах; UNIQUE с учётом soft-delete (можно создать с тем же ключом после удаления); FK/джойны на soft-deleted; восстановление; нет «полуудалённых».
- **Уникальность/дубли:** нормализация перед сравнением (trim/lowercase/unicode email/телефона); идемпотентность по бизнес-ключу; дедуп при импорте/батче; гонка до коммита.
- **Индексы/план:** используется ли индекс (EXPLAIN/ANALYZE, нет seq scan на больших); деградация на росте, запросы без LIMIT, N+1 к БД; keyset вместо offset на глубоких страницах; влияние индекса на запись/bloat.
- **Кодировки/коллации:** UTF-8 по всей цепочке (utf8mb4 для эмодзи); коллация/регистр в сравнении/сортировке/уникальности; усечение байты vs символы (не рвать многобайтный).
- **Числа/деньги:** DECIMAL/integer-минорные не float; округление (half-up/banker's) едино; переполнение типа; масштаб DECIMAL(p,s); split без потери копеек.
- **Время/TZ:** хранение в UTC (timestamptz), единый стандарт; конвертация на чтение; DST; точность долей секунды; границы диапазонов (включ./исключ.); расхождение времени БД и приложения.
- **Ретеншн/GDPR:** удаление/анонимизация по сроку (реально стирается где требует закон); право на забвение из всех мест (основная БД, реплики, кэш, бэкапы, логи, аналитика, индекс); архивация в холодное; каскад анонимизации; аудит удаления.
- **Сидинг/фикстуры:** изоляция тестов (чистое состояние, нет зависимости от порядка); стратегия очистки (rollback/truncate, сброс sequence); фикстуры соответствуют схеме; уникальность данных между параллельными воркерами; детерминизм (фикс seed/время/рандом).

### 3.3 AuthN/AuthZ и мультитенантность
- **AuthN-схемы:** **JWT** (alg=none, RS256→HS256 с публичным ключом, слабый HMAC; проверка `iss`/`aud`/`exp`/`nbf`/`iat`; манипуляция `kid` — traversal/SQLi/SSRF на JWKS); **OAuth2/OIDC** (`state`/CSRF, `nonce`/replay, PKCE; подмена `redirect_uri`, повторное использование code, mix-up); **API-key** (header vs query/утечка в логи, ротация/отзыв, привязка к scope/тенанту); **Basic** (только TLS, brute-force, тайминг сравнения); **mTLS** (срок/CA-цепочка/revocation, CN/SAN-привязка, downgrade).
- **Жизненный цикл токена:** access не «вечный» (просроченный → 401 не 200); refresh rotation + reuse-detection (украденный refresh → инвалидация семьи); отзыв/blacklist (logout реально инвалидирует на сервере, не только клиент); смена пароля/блокировка инвалидирует сессии; clock skew между сервисами.
- **AuthZ:** проверка на **КАЖДОМ** эндпоинте, не только UI/gateway (defense in depth); RBAC (без роли, несколько ролей, удаление роли в сессии); ABAC/scope (недостаточный → 403, чтение не даёт запись); 401(не аутентифицирован) vs 403(нет прав); метод-специфичная (GET да, POST/PUT/DELETE нет; HEAD/OPTIONS не обходят); авторизация на вложенных (`/orders/{id}/items` — и order, и item); default-deny.
- **Эскалация:** вертикальная (admin-эндпоинт под юзером, `role`/`isAdmin` в PATCH, mass-assignment); горизонтальная (чужой ресурс того же уровня); через смену пароля/email чужого; privilege persistence (понижение роли не отзывает токен).
- **IDOR/BOLA:** доступ к чужому объекту по `id` (последовательные/предсказуемые) на чтение/изменение/удаление; UUID ≠ авторизация; IDOR в path/query/body/headers/cookie/вложенных JSON; IDOR на запись (`owner_id`/`tenant_id` чужого); в экспортах/отчётах/вебхуках/файлах; косвенный (фильтры/batch возвращают чужое); BOPLA (свойства объекта не для этой роли).
- **Мультитенантность:** полная изоляция A↔B; tenant из **токена**, не из клиентского параметра (`X-Tenant-Id` подменяем); конфликт tenant в токене ≠ URL → отказ; WHERE tenant_id во всех запросах (утечка через JOIN/агрегаты/отчёты); общий кэш с tenant в ключе; cross-tenant через связанные сущности/вебхуки/джобы/экспорт; изоляция файлов/бакетов; поиск не пересекает границы.
- **Негативные состояния токена:** отсутствует → 401; просрочен → 401; подделан/повреждён (битый base64, лишние сегменты, `Bearer ` без значения) → 401; неверная схема/регистр; токен в query вместо header; токен другого окружения/audience; очень длинный (DoS).
- **Replay/CSRF/brute-force:** replay валидного запроса (платежи) — nonce/idempotency-key; replay токена с другого IP/UA (binding); CSRF на cookie-сессиях (токен/`SameSite`+`Origin`/`Referer`, override через `_method`); rate-limit на login/refresh/reset по аккаунту И по IP, lockout без DoS, защита от `X-Forwarded-For` спуфинга; account enumeration (одинаковый ответ/тайминг); лимит OTP/2FA.
- **Забываемые эндпоинты:** без авторизации совсем (admin/actuator/metrics/health с PII, swagger, GraphQL introspection, `/internal/*`); технические (экспорт/импорт/массовые/webhooks-receiver); версионные дубли (`/v1` закрыт, legacy открыт); прямой доступ в обход gateway; утечки токенов/PII в логах/ошибках; gRPC/очереди/GraphQL-мутации — та же авторизация.

### 3.4 Async / очереди / события / джобы
- **Producer:** сообщение после коммита бизнес-транзакции (не до/не фантомно); корректный topic/routing/partition key; схема (Avro/Protobuf/JSON Schema, версия); заголовки (correlation/trace/message-id/timestamp); поведение при недоступном брокере; publisher confirms/acks/nack; размер > лимита брокера (claim-check).
- **Consumer:** обработка ровно один раз логически; ack только после успеха (поведение при падении до ack); manual vs auto-ack, prefetch; невалидный payload не роняет консьюмер; неизвестный тип/версия → graceful skip или DLQ; обработка дольше visibility timeout → повторная выдача/дубль.
- **Гарантии/идемпотентность:** at-least-once → консьюмер идемпотентен (dedup по message-id/business-key, upsert); exactly-once (чем обеспечен — outbox/Kafka EOS, не маркетинг); повторная доставка не создаёт дублей в БД/побочках (письма/платежи); окно дедупликации (TTL) и поведение после.
- **Порядок:** глобально/в партиции/в ключе; один business-key в одну партицию; out-of-order (created раньше updated, delete раньше create); ребаланс/смена числа партиций; concurrency>1 ломает порядок — допустимо ли; версия/секвенс для отбрасывания устаревших.
- **Ретраи/DLQ:** ретраятся только транзиентные (бизнес-ошибки сразу в DLQ); экспонента+jitter, max attempts; счётчик сохраняется между ретраями; в DLQ — payload+причина+stacktrace+attempt; redrive из DLQ без повторного падения; DLQ мониторится/алертит; ретрай не дублирует побочки.
- **Poison messages:** всегда падающее не блокирует партицию бесконечно; нет infinite retry loop; изоляция в DLQ; head-of-line blocking.
- **Дубли:** от ретраев продюсера/консьюмера, ребаланса, redelivery после crash; дедуп брокера vs приложения; двойная обработка вебхука/платежа; несколько инстансов консьюмера.
- **Eventual consistency:** время сходимости (read-after-write может вернуть старое); тест — polling до условия, не фиксированный `sleep`; SLA лага; реплика отстаёт; частичная сходимость (одна проекция обновилась, другая нет).
- **Саги/outbox:** happy path всей цепочки; падение на каждом шаге → компенсации предыдущих; компенсация идемпотентна; компенсация невозможна (письмо отправлено, деньги выведены); таймаут шага (нет вечного in-progress); двойной запуск по business-key; outbox в той же транзакции, не публикует при откате, порядок=записи, очистка таблицы; inbox для дедупа входящих.
- **Вебхуки:** доставка (2xx=success); подпись (HMAC) проверяется; replay-защита (timestamp+nonce, окно); ретраи при 5xx/таймауте + backoff + DLQ; идемпотентность приёма (event-id не дважды); таймаут на ответ endpoint; out-of-order устойчивость; дубли (дедуп обязателен); недоступный/протухший URL; mTLS/IP allowlist; версионирование payload.
- **Шедулеры/cron:** запуск в назначенное время (cron-выражение); TZ/DST (пропуск или двойной запуск в 02:00–03:00); пропущенный запуск (catch-up vs skip); overlap (lock/skip/queue); single-execution в кластере (distributed lock/leader election); падение (алерт/ретрай); идемпотентность; долгая джоба не выходит за окно.
- **Long-running:** прогресс (queued/running/done/failed, %/этапы); отмена в процессе (реальная остановка, освобождение ресурсов); таймаут (убивается, не висит); рестарт воркера (чекпоинт/безопасный рестарт); частичный результат; повторный сабмит того же job-id.
- **Потеря/задержка:** потеря сообщения детектится (auto-ack до обработки); durability/реплицирование; растущий consumer lag (метрика/алерт); backpressure (продюсер быстрее консьюмера); TTL сообщения; переполнение очереди (отказ/вытеснение).
- **Оснастка:** end-to-end трассировка (correlation сквозь продюсер→брокер→консьюмер→side effects); метрики (lag, размер DLQ, rate ретраев); тест ждёт через polling; изоляция (уникальные топики/очереди на тест-ран, очистка offset/queue/DLQ); fake clock для шедулеров; возможность форсить дубль/переупорядочивание/падение консьюмера в тесте.

### 3.5 Нефункциональное и устойчивость
- **Нагрузка:** load (SLA при целевом RPS, latency p50/p95/p99/p999, error rate, насыщение CPU/RAM/IO); stress (knee point, деградация без падения, 429/503 вместо 5xx); soak/endurance (утечки памяти/коннекшенов/дескрипторов, рост GC-пауз); spike (рост x10, скорость восстановления). Метрики — **percentile, не average**; high-percentile хвосты; coordinated omission (open vs closed model); прогрев перед замером; реалистичный mix запросов; think-time.
- **Масштабируемость:** горизонтальное масштабирование реально повышает throughput; stateless (любой инстанс — любой запрос); шардирование (нет hot partition/key); автоскейл (порог, скорость, нет флаппинга, thundering herd); узкое место (БД pool/блокировки, кэш, внешний API); балансировщик (равномерность).
- **Деградация/пределы:** back-pressure vs неконтролируемое падение; лимиты (max размер запроса/ответа, max соединений/очереди, pool exhaustion); rate limiting (счётчики, fixed/sliding окно, 429+Retry-After, изоляция по клиенту/тенанту); quota; приоритизация (важный трафик не вытесняется, load shedding).
- **Устойчивость:** таймауты на всех вызовах (connect/read/write/total, каскад ≤ родительского); ретраи только идемпотентные (backoff+jitter, нет retry-шторма, budget); circuit breaker (открытие по порогу, half-open, восстановление); bulkhead (изоляция пулов); backpressure; **edge:** ретрай поверх таймаута → дубли (нужна идемпотентность); вложенные ретраи на каждом слое (retry amplification); таймаут клиента < сервера (сервер делает бесполезную работу).
- **Идемпотентность под повтором:** idempotency-key (повтор=прежний результат); дубли из at-least-once очередей (exactly-once на эффекте); повтор после таймаута (нет двойного списания); конкурентный повтор ключа (гонка); TTL ключа; частичный сбой (применилось, ответ не дошёл → повтор завершает корректно).
- **Отказ зависимостей:** БД недоступна (ошибка vs зависание, readiness уводит из ротации); кэш недоступен (fallback на источник, защита от stampede); сторонний API (таймаут, fallback, деградация фичи); очередь/брокер; **частичная деградация** (медленная зависимость — самый коварный кейс); graceful degradation (отключение второстепенного, сохранение ядра); **edge:** DNS-сбой, TLS-handshake fail, истёкший сертификат, 200 с битым телом, флапающая зависимость.
- **Chaos / fault injection:** kill -9 инстанса (нет потери данных/застрявших транзакций); сетевые сбои (потеря пакетов, partition/split-brain); деградация ресурса (CPU throttle, заполнение диска, OOM, inode); latency/error injection на % трафика; clock skew/прыжок NTP; game day/проверка runbook; одновременный отказ нескольких зависимостей; отказ лидера (failover).
- **Восстановление:** рестарт (восстановление состояния, drain in-flight); после краша (незавершённые транзакции откат/докат, нет «осиротевших»); recovery после возврата зависимости (переподключение пула); cold start (порядок старта, нет deadlock); реконсиляция отложенных событий; RTO/RPO соответствуют целевым; **edge:** рестарт под нагрузкой; одновременный рестарт всех инстансов; потеря лидера во время длинной операции.
- **Утечки ресурсов:** память (heap без возврата под soak, native, кэш без TTL); connection pool (возврат при исключении, idle-таймауты); file descriptors/сокеты (закрытие после ошибки, TIME_WAIT); потоки/горутины (зависшие воркеры, thread pool exhaustion); unbounded queue → OOM; временные файлы/ротация логов; **edge:** утечка только на ошибочном пути / при отмене запроса клиентом / медленная (видна за часы).
- **Наблюдаемость (как тестируемое свойство):** структурные логи (JSON, уровни, без PII/секретов, контекст, логирование на всех catch); метрики RED (Rate/Errors/Duration) + USE (Utilization/Saturation/Errors), кардинальность лейблов; распределённая трассировка (span на вызов, propagation сквозь сервисы и очереди); correlation-id (генерация на входе, проброс во все downstream и логи, возврат клиенту); алерты на симптомы (SLO burn rate), runbook на каждый, проверка что реально срабатывает; **edge:** логи под нагрузкой не узкое место, потеря трейса через очередь.
- **Health-пробы:** liveness (жив ли процесс, не зависит от внешних зависимостей — иначе каскадный рестарт); readiness (готовность зависимостей, уводит из ротации при деградации); startup-проба для медленного старта; **edge:** readiness на необязательной зависимости (ложный вывод из ротации); «зелёный» health, а сервис не обслуживает (поверхностная проба); flapping readiness.
- **Конфиг/секреты/флаги:** валидация конфига на старте (fail-fast, типизация, precedence CLI>ENV>file>default); разделение окружений, нет prod-секретов в репо/образе/логах; ротация секретов без рестарта (окно валидности двух ключей); hot reload конфига (атомарность); фичефлаги (обе ветки, дефолт при недоступности сервиса флагов — fail-secure, sticky bucketing, kill-switch мгновенно, флаг не единственный барьер авторизации, нет рассинхрона между репликами).
- **Zero-downtime деплой:** rolling update (старая+новая версии одновременно, контракт совместим N и N+1); schema migration expand-contract (совместима со старым кодом, обратима); контракт сообщений (tolerant reader, неизвестные поля игнорируются); graceful drain (SIGTERM → дозавершение in-flight, terminationGracePeriod); быстрый rollback без потери данных (необратимые миграции выделены); canary/blue-green (доля трафика, авто-rollback по метрикам); **edge:** удаление поля до обновления клиентов; порядок выкатки БД и кода; long-running запрос пересекает деплой; in-flight сообщение в старом формате; смешанный кластер версий дольше ожидаемого.
- **Контейнеры/k8s:** liveness/readiness/startup; graceful shutdown (дренаж); stateless в N репликах (нет локального стейта/файлов); resource limits (OOMKill, CPU throttling); rolling/canary; config/secrets через ConfigMap/Secret; DNS-резолв/таймауты внутри кластера; persistent volumes; HPA триггерится без crash loop; TZ/локаль контейнера (UTC), CA в образе; non-root, read-only FS.

### 3.6 Security backend (OWASP API Top 10)
- **1. Injection:** SQL/NoSQL в каждом параметре/заголовке/cookie (`' OR '1'='1`, `$ne`/`$where`); слепые (time-based `SLEEP`, boolean, OOB DNS/HTTP); второго порядка (срабатывает при чтении); command/OS (`; | $(cmd) &&`); LDAP; XPath/XXE/SSTI (`{{7*7}}`/`${...}`); ORM/GraphQL; **edge:** ORDER BY/LIMIT/имена колонок (не параметризуются), LIKE-маски, закодированные payload, numeric-контекст без кавычек.
- **2. BOLA/IDOR:** подмена `id`/`uuid`/`slug` в path/query/body → 403/404 не 200; перебор последовательных и утёкших UUID; вложенные ресурсы; запись/удаление чужого (не только чтение); через batch/GraphQL node/экспорт; **edge:** косвенное владение (заказ→платёж→документ) — проверка на каждом уровне цепочки.
- **3. Broken Authentication:** brute-force login/OTP без блокировки/rate-limit, credential stuffing; слабый пароль, нет проверки на утёкшие; JWT (`alg:none`, RS256→HS256, слабый секрет, нет проверки exp/iss/aud); нет инвалидации при logout/смене пароля; user enumeration (разные ошибки/тайминги); сброс пароля (предсказуемый/долгоживущий токен, host-header injection); **edge:** OTP без TTL/лимита, обход 2FA на втором шаге.
- **4. BOPLA:** mass assignment (`role`/`isAdmin`/`balance`/`status`/`price` в body на create/update → игнор; через вложенные/массивы/alias/PATCH); excessive data exposure (`passwordHash`/`internalNotes`/чужой email/токены в ответе — фильтрация на сервере, не клиенте; утечка через `include`/`expand`/`fields`/error-объекты/debug); **edge:** разные роли — разный набор полей (матрица роль×поле), list отдаёт больше чем detail.
- **5. BFLA:** admin-эндпоинты под юзером (`/admin/*`, `/internal/*`, DELETE/PUT); подмена метода (override `X-HTTP-Method-Override`); функции чужой роли по горизонтали; скрытые/недокументированные эндпоинты (из JS/swagger/старых версий); **edge:** авторизация только на UI/gateway, не на сервисе; альтернативный маршрут (gRPC/internal).
- **6. Unrestricted Resource Consumption:** нет rate-limit на login/OTP/поиск/экспорт/регистрацию/email-SMS; обход лимита (`X-Forwarded-For`, UA, кейс email, `+alias`); дорогие операции (большие `limit`, рекурсивный GraphQL, batch без лимита); нет лимитов на размер body/файла/числа полей (DoS, биллинг-абуз); **edge:** ZIP-бомба, ReDoS (катастрофический regex), сложность по CPU/памяти.
- **7. SSRF:** любой параметр-URL (webhook/callback/import-by-url/аватар/превью) → `http://169.254.169.254/` (cloud metadata), localhost, internal-IP, `file://`/`gopher://`; обход фильтров (DNS rebinding, редирект на internal, IP в decimal/octal/hex, `[::1]`, `http://allowed@internal`); blind SSRF (таймаут/OOB); **edge:** через парсинг SVG/XML/PDF, через webhook-валидацию.
- **Десериализация:** небезопасная (Java gadget chains, Python pickle, PHP unserialize, YAML `!!python/object` → RCE); polymorphic (Jackson `@type`); **edge:** из cookie/кэша/очереди/Redis.
- **8. Security Misconfiguration:** заголовки (нет HSTS/nosniff/CSP/X-Frame-Options, нет `Cache-Control:no-store` на чувствительных, есть `Server`/`X-Powered-By`); CORS (`*`+credentials, рефлексия Origin, `null`-origin, регэксп-обход, поддоменные дыры); TLS (HTTP→HTTPS редирект, слабые ciphers TLS1.0/1.1, валидность сертификата, mixed content); шифрование at-rest (PII/токены в открытом виде, нехешированные пароли или MD5/SHA1 без соли); дефолтные креды, открытый actuator/`/debug`/`/env`/swagger в prod, directory listing, verbose-ошибки.
- **9. Improper Inventory Management:** живые deprecated версии (`/v1`, beta, staging с prod-данными); недокументированные/debug-эндпоинты, забытые тестовые аккаунты; несоответствие реализации и swagger; **edge:** одинаковые данные через несколько API с разной защитой.
- **10. Unsafe Consumption of 3rd-party:** доверие к ответам внешних API без валидации (инъекция/SSRF через интегратора); нет проверки TLS/подписи; зависимости с CVE (`npm audit`/`pip-audit`/Trivy/Snyk, SCA в CI); **edge:** уязвимая зависимость в одном микросервисе, lock-файл не отражает реальные версии.
- **Кросс-функциональное:** валидация/санитизация входа (тип/формат/диапазон/длина/enum, null-byte `%00`, CRLF в заголовки/логи, канонизация перед валидацией, path traversal `../`/`..%2f`); загрузка файлов (magic bytes не extension, double-extension `.jpg.php`, лимит до и после распаковки, ZIP-бомба, антивирус/EICAR, хранение вне webroot, рандомизация имён, SVG/polyglot с XSS/SSRF); секреты/утечки (в коде/git-истории/Docker-слоях — trufflehog/gitleaks; в логах/ответах/error/URL/JWT-payload); аудит-логи (вход/выход, смена прав, доступ к PII, финансовые, провалы авторизации — кто/что/когда/откуда, неизменяемость); гонки/TOCTOU (двойное списание, обход лимита, повторное использование купона); кэш-отравление; 401/403/404 не раскрывают существование.
